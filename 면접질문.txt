> ERP 
    - Enterprise Resource Planning의 약어, 기업 자원 관리를 의미
    - 기업 내부에서 비즈니스 프로세스를 관리하고 통합하는 소프트웨어 시스템
    - 기업 내의 다양한 부서와 작업들이 긴밀하게 연결되어 있기 때문에, 각각의 부서에서 이뤄지는 업무를 통합하고, 비즈니스 프로세스를 효율적으로 관리하고자 ERP 시스템이 사용됨
    - 한 번의 데이터 입력으로 여러 부서에서 공유하고 사용할 수 있는 데이터베이스를 기반으로함
    - 업무의 효율성을 높이고, 비용을 절감하며, 의사 결정에 필요한 신뢰할 수 있는 데이터를 더 빠르게 얻을 수 있음

> 객체지향
    - 프로그램을 객체들의 모음으로 구성하는 방식을 강조하는 프로그래밍 언어, 데이터와 그 데이터를 처리하는 메서드(함수)로 구성
    - 코드의 재사용성, 유지보수성, 확장성 등이 개선되며, 복잡한 문제를 더 간결하게 해결
    - 단점
        - 현실 세계의 개념을 모델링하기 위해 추상화, 상속, 다형성 등의 개념을 사용
            - 이로 인해 프로그램의 구조가 복잡해질 수 있음, 복잡성이 증가하면 이해하고 유지보수하기 어려워질 수 있음
        - 절차지향 프로그래밍보다 더 많은 메모리와 처리 시간을 필요로 함, 객체 간의 상호작용이 많은 경우 성능 저하가 발생할 수 있음

> 절차지향
    - 프로그램을 일련의 절차나 순차적인 단계로 구성
    - 순차적인 실행이 강조되며, 데이터와 기능(함수)을 분리하여 다룸
    - 프로그램의 규모가 커지고 복잡해질수록 코드의 가독성과 유지보수의 어려움이 증가함

> 백엔드
    - 사용자가 직접적으로 접하는 부분이 아닌, 서버 측에서 데이터 처리, 비즈니스 로직, 데이터베이스 관리 등을 담당하는 부분을 의미

> 프론트엔드
    - 사용자가 직접적으로 상호작용하고 시각적으로 볼 수 있는 부분을 의미

> 돔
    - 웹 페이지의 구조를 표현하는 객체 지향적인 프로그래밍 인터페이스
    - 웹 페이지의 HTML, XML, 또는 XHTML 문서를 트리 형태의 객체로 변환하여 자바스크립트와 같은 스크립트 언어에서 문서의 요소를 동적으로 접근하고 조작할 수 있게 해줌

> 가상돔
    - 사용자 인터페이스(UI)의 성능을 향상시키기 위한 기술. 웹 애플리케이션에서는 사용자의 동작에 따라 화면이 동적으로 변경되는데, 이러한 UI 업데이트를 효율적으로 처리하기 위해 가상돔이 사용됨
    - 가상돔은 메모리에 가상으로 존재하는 가벼운 복사본으로서, 실제 DOM과 동일한 구조를 가짐
    - 웹 페이지의 상태가 변경될 때마다 가상돔을 사용하여 원본 DOM과 비교하고, 변경된 부분만을 실제 DOM에 반영함.
    - 변경된 부분만을 업데이트하는 것으로 불필요한 DOM 조작을 최소화하고, 성능을 향상시킬 수 있음.

> CSRF
    - Cross-Site Request Forgery, 인증된 사용자의 웹 애플리케이션을 통해 악의적인 요청을 위조하는 공격
    - 사용자의 동의나 인지 없이 공격자가 의도한 액션을 수행하도록 유도

> MVC
    - Mode-View-Controller, 소프트웨어 개발에서 사용되는 디자인 패턴, 애플리케이션의 구조를 세 가지 역할로 분리하여 개발하는 방법
    - 각 역할은 애플리케이션 내에서 특정한 책임을 가지며, 서로의 영향을 최소화하여 유지보수와 확장을 용이하게 함
    - Model
        - 애플리케이션의 데이터와 비즈니스 로직을 담당
        - 데이터를 관리하고, 데이터의 유효성 검사 및 조작, 데이터베이스와의 상호작용 등을 담당
    - View
        - 사용자에게 데이터를 시각적으로 표현하는 역할을 담당
        - 모델의 데이터를 표시하는 부분이며, 사용자 인터페이스(UI)를 구성하는 역할
    - Controller
        - 사용자의 입력을 처리하고, 모델과 뷰를 연결하는 역할
        - 사용자의 요청을 받아 해당하는 모델을 호출하여 데이터를 처리하고, 뷰를 업데이트함
    - MVC 패턴을 사용하면 데이터와 비즈니스 로직을 별도로 유지하므로 코드의 재사용성이 높아지고, 애플리케이션의 유지보수와 확장이 편리해짐
    - 각 역할이 분리되어 개발을 병렬적으로 진행할 수 있음
